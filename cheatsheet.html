Part 1 — First-Time Setup · Run Once Per Project
1
Install Dependencies — one time, per machine
Python (backend parser + watcher)
pip install tree-sitter watchdog

Powers Tree-sitter PHP/TS parsing and the live file watcher.

Node.js (alternative watcher)
npm install chokidar

Only needed if you prefer a Node watcher over Python's watchdog. The graph logic is identical.

Tree-sitter Grammars — Build Once Per Project
# PHP grammar (Laravel backend)
Language.build_library('.codebase-graph/parsers/php.so', ['vendor/tree-sitter-php'])

# TypeScript / TSX grammar (Next.js frontend)
Language.build_library('.codebase-graph/parsers/tsx.so', ['vendor/tree-sitter-typescript/tsx'])

These are compiled once and cached in .codebase-graph/parsers/. You don't rebuild them unless you update the grammar source.

2
Run Init — once per project (or after major restructures)
Dry run first — always a good idea
/codebase-graph init --dry-run   # shows detected stack + plan, writes nothing
Full-stack project
/codebase-graph init

Runs all 4 phases: backend → frontend → API bridge → clusters + PageRank. Takes 3–10s.

Backend or frontend only
/codebase-graph init -b   # Laravel only
/codebase-graph init -f   # Next.js only
⚠

Phase order is mandatory and cannot be skipped. Each phase depends on the nodes from the previous one. Migrations → Models → Controllers → Routes → Components → Pages → Bridge → Clusters/PageRank.

When to re-run init
Situation Command
Pulled massive refactor / rename (renamed dirs, restructured app) /codebase-graph init --reset
Added a whole new module or domain (e.g. new payment system) /codebase-graph init --reset
Graph seems wrong, clusters look off /codebase-graph init --reset
Normal daily work, watcher running No init needed — watcher handles it
Part 2 — Starting Every Work Session · 2 Commands
1
Start the Watcher — first thing you do when you open the project
Background (recommended)
/codebase-graph watch --daemon

Runs silently, writes PID to .codebase-graph/watch.pid. Forget about it for the rest of the session.

Other watcher commands
/codebase-graph watch --status   # is it running?
/codebase-graph watch --stop     # end of session
/codebase-graph watch            # foreground (debug)
🔴

If the watcher is not running, the graph goes stale. Every read command will warn you. Every blast-radius report will be based on old data. Always check --status if something feels off.

The watcher monitors .php .ts .tsx .js .jsx and ignores vendor/ node_modules/ .next/ .codebase-graph/. On every file save it deletes stale nodes, re-parses the file, re-inserts fresh nodes/edges, and re-maps API bridge for that file — all in a single atomic transaction.

2
Load Context — before your first prompt of the session 
/codebase-graph read   # full architecture overview — your starting point

This loads the structured context block into the AI's working memory: all clusters with their models/controllers/routes/frontend nodes, the top-20 PageRank hotspots, and graph stats (node count, bridge edges, dead candidates, watcher status). Use this to orient yourself before starting work.

Focused alternatives — use when you already know where you're working
/codebase-graph read --file app/Models/User.php     # subgraph around one file, depth 2 both directions
/codebase-graph read --query "authentication"       # search nodes by name/path/metadata
/codebase-graph read --cluster 3                    # everything in one architectural cluster
/codebase-graph read --hotspots                      # top-20 PageRank nodes only
💡

Which read to use when: Start with plain /codebase-graph read for orientation. Switch to --file once you know what you're touching. Use --query when you're exploring a domain you're unfamiliar with. Use --cluster when you're doing deep work in one subsystem all session.

Part 3 — Every Code Change · The Core Loop
→
The Mandatory 3-Step Loop — every time you're about to modify a file

Read context for the specific file you're about to touch

/codebase-graph read --file app/Models/User.php

This gives you the seed node in full detail + everything upstream and downstream at depth 2. Confirms what the file does, what it depends on, and what depends on it — before you touch anything.

Run blast radius to see the full impact

/codebase-graph blast-radius --file app/Models/User.php

Traverses outward up to depth 5. Produces a ranked list of every file that may break. Read it as a review checklist — not a to-do list. Not all affected files need to change; they need to be verified.

Make your change — the watcher handles the rest

Save the file. The watcher detects the change, re-parses the file, updates nodes/edges in a single transaction, and re-maps API bridge calls for that file. The graph is live again within milliseconds.

▸

Risk levels
Level Meaning
CRITICAL The file you changed
HIGH Depth 1 or edge weight ≥ 8
MEDIUM Depth 2 or weight 5–7
LOW Depth 3+ and weight < 5
Edge weights — what bleeds impact
Weight Edge
10 BACKED_BY (model→migration)
9 HAS_MANY / BELONGS_TO
8 HANDLED_BY / CALLS_API
7 QUERIES
6 RENDERS
5 USES_HOOK
4 DEPENDS_ON
3 PROTECTED_BY / OBSERVED_BY
Blast radius flags — when to use each
/codebase-graph blast-radius --file X                   # standard — downstream only, depth 5
/codebase-graph blast-radius --file X --method store   # one method only — much tighter scope
/codebase-graph blast-radius --file X --direction both  # upstream too — use when investigating bugs
/codebase-graph blast-radius --file X --depth 3         # shallower — for quick sanity checks
Part 4 — Change-Type Playbooks
Changing a Migration / Schema

Highest possible blast radius. A schema change can ripple through model → controllers → routes → frontend.

read --file database/migrations/…
blast-radius --file database/migrations/…
# Check the BACKED_BY model + all QUERIES controllers

After adding a column: also update $fillable / $casts on the model, any Resources that serialize it, and any frontend types that mirror the shape.

Changing a Model

Adding/removing relationships, scopes, or fillable fields. 

read --file app/Models/User.php
blast-radius --file app/Models/User.php
# High-risk: any controller with QUERIES edge
# Also check: Resources, Observers, Policies
Changing a Controller Method

Method-level blast radius keeps scope tight.

blast-radius --file app/Http/Controllers/UserController.php --method index
# Checks: route → frontend hook → components
# If changing response shape, check UserResource too
Changing a Frontend Component

Usually bounded within the frontend cluster unless CALLS_API edge exists.

read --file src/components/UserCard.tsx
blast-radius --file src/components/UserCard.tsx
# Look for RENDERS edges — parent components
# If props change, find every RENDERS → UserCard
Changing a Hook with API Calls

Hooks with CALLS_API edges bridge into the backend graph. 

read --file src/hooks/useUsers.ts
blast-radius --file src/hooks/useUsers.ts --direction both
# upstream: which backend route/controller it calls
# downstream: which components use this hook
Changing a Route File

URI changes break the API bridge. Verb changes break REST contracts.

index-bridge --report   # after any route change
# Check: any unmapped frontend calls = broken
# Re-run index-bridge if watcher misses a batch
Part 5 — Long Sessions & Token Management
△
Staying Efficient Across Multiple Prompts
Context budget strategy
Situation Command Tokens used
Start of session, orientation /codebase-graph read ~4 000 (default)
Focused work on one subsystem /codebase-graph read --file X ~1 500–2 500
Deep multi-file session in one domain /codebase-graph read --cluster N ~2 000–3 000
Tight token budget /codebase-graph read --budget 1500 ≤ 1 500 hard cap
Just want the hottest files /codebase-graph read --hotspots ~300–500
Multi-prompt: use incremental diff

On the second+ prompt in a session, the graph uses a context diff — only sending what changed since the last read. This saves 60–80% of tokens per prompt. You don't need to do anything; it's automatic.

CONTEXT DELTA since last read:
+ UserProfile [component]  src/components/UserProfile.tsx
+ useProfile [hook]        src/hooks/useProfile.ts
- OldDashboard [page]      removed (file deleted)
= 47 nodes unchanged
When to re-read mid-session
  • ↻ You switched to a completely different domain (different cluster)
  • ↻ You just added a new file (watcher indexed it, but context hasn't refreshed)
  • ↻ You did a big rename and the graph now has new node names
  • ↻ The AI gave you a file path that doesn't match the graph — means stale context
Part 6 — Partial Re-Indexing · When the Watcher Isn't Enough
ℹ

The watcher handles individual file saves automatically. You only need partial re-index commands when you've made changes outside normal editing — bulk git operations, code generation, scaffolding, or after the watcher was off for a while.

Partial re-index commands
# Backend — re-index just what changed
/codebase-graph index-backend --step migrations     # after running php artisan make:migration
/codebase-graph index-backend --step models          # after make:model
/codebase-graph index-backend --step controllers     # after make:controller
/codebase-graph index-backend --step routes          # after editing web.php / api.php
/codebase-graph index-backend                        # full backend re-index

# Frontend — re-index just what changed
/codebase-graph index-frontend --step components     # after adding many components at once
/codebase-graph index-frontend --step pages           # after adding new pages
/codebase-graph index-frontend --step hooks           # after adding/renaming hooks
/codebase-graph index-frontend                        # full frontend re-index

# API bridge — re-map after route or fetch changes
/codebase-graph index-bridge                          # re-map all frontend → backend calls
/codebase-graph index-bridge --report               # show unmapped calls + dead routes
Partial re-index order matters

If you re-index multiple steps, always follow the same phase order. Don't re-index controllers before models. Don't re-index pages before components. Don't re-run bridge before both backend and frontend are current.

After scaffolding (make:model -mcr)

Laravel's artisan scaffolding creates migration + model + controller in one shot. Run:

index-backend --step migrations
index-backend --step models
index-backend --step controllers
index-backend --step routes

Part 9 — The Rules You Can't Break
  • 1 Always run blast-radius before modifying a file. Without it you're flying blind. The graph exists for this — use it every time, even when the change "seems small".
  • 2 Always run read before every code-touching prompt. The graph is the source of truth, not training memory. File paths, class names, relationships — confirm everything from the graph.
  • 3 Migration changes are highest priority blast radius. A schema change can silently break model casts, controller queries, API resources, and frontend type assumptions. Always trace the full chain.
  • 4 The blast radius report is a review checklist, not a to-do list. Affected files don't all need to change — they need to be verified for compatibility with your change.
  • 5 Don't re-init when partial re-index is enough. Reserve init --reset for structural refactors. For normal work, the watcher handles everything. For scaffolding, use index-backend --step.
  • 6 If the watcher warning appears in read output, stop and fix it. Don't continue working with a stale graph — any blast-radius report will be wrong.
  • 7 Use --file read for focused work, plain read for orientation. Don't blast 4000 tokens of full architecture context into every prompt. Match the read scope to what you're actually doing.
  • 8 High-PageRank nodes = high-impact files. Before touching a node with high PageRank, expect a wide blast radius. Extra caution, extra review.
